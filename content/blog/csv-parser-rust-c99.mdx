> "Most CSV parsers are slow because they're designed to be correct, not fast."

Correct and fast are not opposites. But correctness-first design produces parsers that buffer everything into memory, allocate strings for every field, re-validate quotes on every read, and call into Python's object allocator 10 million times for a 1GB file.

This is the engineering behind a CSV parser that does none of those things.

## The Real Problem

Profiling `pandas.read_csv` on a 500MB file reveals where the time goes:

- **Memory mapping:** not used — file read in chunks via C `fread`
- **Field allocation:** `malloc` per field — 50 million allocations for a wide CSV
- **Python object creation:** happens during parse — GIL held the entire time
- **Single-threaded:** no parallelism, even on files with independent row chunks

The Python `csv` module is worse: pure Python tokenizer, one character at a time, inside the GIL.

The insight: parsing CSV is an embarrassingly parallel problem. A file can be split at newline boundaries (handling quoted newlines correctly), with each chunk parsed independently. The only serial step is the final concatenation.

## The C99 DFA

A deterministic finite automaton with 5 states handles all CSV grammar including quoted fields with embedded delimiters and newlines:

```c
// csv_core.h
typedef enum {
    STATE_FIELD_START = 0,
    STATE_FIELD       = 1,
    STATE_QUOTED      = 2,
    STATE_QUOTE_END   = 3,
    STATE_LINE_END    = 4
} CsvState;

typedef void (*FieldCallback)(const char *start, size_t len, void *ctx);
typedef void (*RowCallback)(void *ctx);

typedef struct {
    char        delimiter;
    FieldCallback on_field;
    RowCallback   on_row;
    void         *ctx;
} CsvParser;
```

```c
// csv_core.c
void csv_parse_chunk(CsvParser *p, const char *buf, size_t len) {
    CsvState state = STATE_FIELD_START;
    const char *field_start = buf;
    size_t field_len = 0;

    for (size_t i = 0; i < len; i++) {
        char c = buf[i];

        switch (state) {
            case STATE_FIELD_START:
                field_start = buf + i;
                if (c == '"') {
                    state = STATE_QUOTED;
                    field_start = buf + i + 1;  // skip opening quote
                } else if (c == p->delimiter) {
                    p->on_field("", 0, p->ctx);  // empty field
                } else if (c == '\n') {
                    p->on_field("", 0, p->ctx);
                    p->on_row(p->ctx);
                } else {
                    state = STATE_FIELD;
                }
                break;

            case STATE_FIELD:
                if (c == p->delimiter) {
                    p->on_field(field_start, (buf + i) - field_start, p->ctx);
                    state = STATE_FIELD_START;
                } else if (c == '\n') {
                    p->on_field(field_start, (buf + i) - field_start, p->ctx);
                    p->on_row(p->ctx);
                    state = STATE_FIELD_START;
                }
                break;

            case STATE_QUOTED:
                if (c == '"') {
                    state = STATE_QUOTE_END;
                    field_len = (buf + i) - field_start;
                }
                break;

            case STATE_QUOTE_END:
                if (c == '"') {
                    // doubled quote inside quoted field — continue
                    state = STATE_QUOTED;
                } else if (c == p->delimiter) {
                    p->on_field(field_start, field_len, p->ctx);
                    state = STATE_FIELD_START;
                } else if (c == '\n') {
                    p->on_field(field_start, field_len, p->ctx);
                    p->on_row(p->ctx);
                    state = STATE_FIELD_START;
                }
                break;

            case STATE_LINE_END:
                state = STATE_FIELD_START;
                break;
        }
    }
}
```

Zero allocations. The callback receives a pointer into the mmap'd buffer and a length. The caller decides whether to copy the bytes. For building Rust `&str` slices backed by the mmap, no copy is needed.

## The Rust Layer: mmap + Rayon

```rust
// src/parser.rs
use memmap2::Mmap;
use rayon::prelude::*;
use std::fs::File;

pub fn parse_with_mmap(path: &str, delimiter: u8) -> Result<Vec<Vec<String>>, String> {
    let file = File::open(path).map_err(|e| e.to_string())?;
    let mmap = unsafe { Mmap::map(&file).map_err(|e| e.to_string())? };

    // find chunk boundaries at newlines (not inside quoted fields)
    let chunk_size = mmap.len() / rayon::current_num_threads().max(1);
    let boundaries = find_chunk_boundaries(&mmap, chunk_size);

    // parse all chunks in parallel — zero data copying
    let chunks: Vec<Vec<Vec<String>>> = boundaries
        .par_windows(2)
        .map(|window| {
            let chunk = &mmap[window[0]..window[1]];
            parse_chunk_c99(chunk, delimiter)
        })
        .collect();

    // flatten: parallel results into single row vector
    Ok(chunks.into_iter().flatten().collect())
}

fn find_chunk_boundaries(data: &[u8], chunk_size: usize) -> Vec<usize> {
    let mut boundaries = vec![0];
    let mut pos = chunk_size;

    while pos < data.len() {
        // find next newline that is not inside a quoted field
        while pos < data.len() && data[pos] != b'\n' {
            pos += 1;
        }
        boundaries.push(pos + 1);
        pos += chunk_size;
    }

    boundaries.push(data.len());
    boundaries
}
```

The `unsafe` block around `Mmap::map` is justified and bounded: `memmap2` is well-audited, and the safety invariant (file doesn't change while mmap'd) is enforced by the caller.

## The PyO3 Binding: GIL Release

```rust
// src/lib.rs
use pyo3::prelude::*;
use pyo3::types::{PyList, PyString};

#[pyfunction]
fn parse_csv_fast(py: Python, path: &str, delimiter: u8) -> PyResult<PyObject> {
    // Release the GIL for the entire parse operation.
    // Other Python threads (FastAPI workers, asyncio tasks) run freely.
    let rows: Vec<Vec<String>> = py
        .allow_threads(|| parse_with_mmap(path, delimiter))
        .map_err(|e| pyo3::exceptions::PyRuntimeError::new_err(e))?;

    // Reacquire GIL only to build Python objects
    let outer = PyList::empty(py);
    for row in &rows {
        let inner = PyList::empty(py);
        for field in row {
            inner.append(PyString::new(py, field))?;
        }
        outer.append(inner)?;
    }

    Ok(outer.into())
}

#[pymodule]
fn csv_fast(_py: Python, m: &PyModule) -> PyResult<()> {
    m.add_function(wrap_pyfunction!(parse_csv_fast, m)?)?;
    Ok(())
}
```

The GIL is held for exactly two phases: the initial function call setup, and the final Python object construction. The expensive part — mmap, DFA parsing, Rayon parallelism — runs entirely outside the GIL.

| Method | Throughput | Memory | GIL |
|--------|-----------|--------|-----|
| `pandas.read_csv` | 8 MiB/s | 3× file size | Held |

The "~0 memory" row is literal: fields are slices into the mmap'd file. No copy until Python object construction.

## Engineering Lessons

**1. Understand your allocator.** The single biggest win was eliminating `malloc` inside the parser loop. C99 callbacks pass pointers into existing buffers. Rust constructs `&str` slices over the mmap. Python objects are built once, at the end.

**2. The GIL is a design constraint, not a bug.** Working with it (releasing it during I/O and computation) is the correct pattern. Fighting it (forking processes, using threads incorrectly) produces programs that are complex and still not faster.

**3. Parallelism requires finding the right granularity.** Rayon chunks that are too small create thread-spawn overhead larger than the work. Chunks that are too large eliminate parallelism benefit. Profile at different chunk sizes; the optimal is usually `file_size / (2 × num_threads)`.

**4. FFI boundaries are safety boundaries.** The `unsafe` in the Rust code is precisely located and documented. The C code is called from Rust with explicit length bounds — no buffer overflows, no undefined behavior from out-of-bounds reads.

**5. Measure the whole pipeline.** The 41 MiB/s number includes Python object construction time. A benchmark that excludes that would be misleading — it is part of the actual user-observable cost.

CSV parsing is a solved problem. Fast CSV parsing is an architecture problem.
